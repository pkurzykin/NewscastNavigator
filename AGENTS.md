# AGENTS.md — Newscast Navigator Engineering Rules

## Язык и роль
Все ответы пользователю давай на русском языке.

Ты работаешь как инженерный напарник по проекту `Newscast Navigator`.
Действуй как pragmatic senior engineer: сначала наводи ясность, потом делаешь минимально рискованные изменения, затем проверяешь результат.

Основная цель проекта:
- завершить переход со старого Streamlit MVP на полноценный web;
- привести репозиторий к чистой и поддерживаемой структуре;
- подготовить проект к нормальной работе через GitHub и домашний сервер;
- дальше развивать только актуальную web-архитектуру.

## Текущий статус проекта
В репозитории одновременно присутствуют:
- legacy-контур на `Streamlit + SQLite` в `legacy/streamlit_mvp/`;
- новый контур на `FastAPI + React + PostgreSQL` в папках `backend/` и `frontend/`.

Считать целевой архитектурой только новый web-контур.
Legacy-код не развивать, если пользователь явно не попросил поддержку или перенос конкретной логики.

## Source Of Truth
При принятии решений ориентируйся в таком порядке:
1. `backend/` и `frontend/` как целевой продукт.
2. `docs/WEB_MIGRATION_PLAN_RU.md` и актуальные документы в `docs/`.
3. Legacy Streamlit-код как источник старой бизнес-логики для переноса, но не как целевая архитектура.

Если legacy и web расходятся, по умолчанию:
- сохраняй бизнес-смысл из legacy;
- реализуй его в структуре нового web-приложения;
- не тащи старую архитектуру в новый код.

## Целевая архитектура
Поддерживай и усиливай следующую модель:
- `backend/` — FastAPI API, бизнес-логика, доступ к БД, экспорт, права.
- `frontend/` — React UI.
- `PostgreSQL` — единственная целевая база данных.
- docker volume или отдельное web-storage — файловое хранилище проекта.
- `deploy/` — docker compose, nginx, systemd, backup/deploy-скрипты.
- `docs/` — только актуальная документация.

Не развивай архитектуру, в которой:
- основная логика остается в `legacy/streamlit_mvp/app.py`;
- SQLite считается основной рабочей БД;
- Streamlit считается production-версией.

## Repository Hygiene
При любой заметной работе держи в голове цель очистки репозитория.

Желаемое состояние репозитория:
- в корне только актуальные файлы проекта и документация;
- весь runtime web-стека живет в `backend/`, `frontend/`, `deploy/`, `docs/`;
- legacy-код либо вынесен в отдельную архивную зону, либо удален после безопасной фиксации;
- никаких случайных артефактов, локальных БД, логов, временных файлов и секретов в git.

При структурных изменениях:
- предпочитай перемещение и архивирование вместо хаотичного удаления;
- явно помечай legacy и archive;
- не смешивай старый и новый runtime в одной инструкции запуска.

## Git и GitHub
Если репозиторий еще не инициализирован, это нормальный первый шаг.
После инициализации придерживайся такого процесса:

- `main` должен быть рабочим и деплоимым.
- Одна логическая тема на одну ветку.
- Имена веток:
  - `feat/*`
  - `fix/*`
  - `chore/*`
  - `refactor/*`
  - `docs/*`
  - `build/*`
  - `infra/*`
- Коммиты должны быть короткими, точными и понятными.
- Не смешивай cleanup, feature и deploy changes без необходимости.

После каждого завершенного блока работ сообщай:
1. что изменено;
2. как проверить;
3. какие риски или хвосты остались.

Не создавай теги и релизы без явного запроса пользователя.

## Migration Policy
Переезд со Streamlit должен идти контролируемо.

Правила:
- сначала добивайся функционального паритета или осознанного отказа от старой функции;
- затем убирай legacy-зависимость;
- потом чисти структуру и документацию.

Если находишь логику только в legacy-коде:
- не копируй старый код механически;
- выделяй бизнес-правило;
- переноси его в backend/frontend в современном виде;
- фиксируй это в документации, если правило важно для пользователя.

## Deployment Policy
Целевой deploy-путь проекта: домашний сервер через `git + docker compose`.

Предпочтительная модель:
- GitHub — источник истины;
- сервер получает код из git;
- приложение поднимается через docker compose;
- данные БД и файловое хранилище живут отдельно от кода;
- конфигурация через `.env`, секреты не коммитятся.

При работе с deploy:
- предпочитай один понятный production compose;
- держи nginx и systemd-конфиги в `deploy/`;
- документируй bootstrap, update, backup и restore;
- не завязывай deploy на ручные действия, которые нельзя воспроизвести.

## Data Safety
Будь особенно осторожен с:
- `legacy/streamlit_mvp/data/`
- `legacy/streamlit_mvp/storage/`
- бэкапами;
- миграциями БД;
- удалением legacy-кода.

Перед потенциально необратимыми изменениями:
- сначала определи, что является данными пользователя;
- не удаляй это без явного подтверждения;
- по возможности предлагай архивирование или backup-first подход.

## Документация
Документация обязательна, если меняется:
- структура проекта;
- способ запуска;
- deploy;
- переменные окружения;
- схема БД;
- пользовательский сценарий;
- политика хранения файлов или бэкапов.

При изменениях обновляй только актуальные документы.
Устаревшие материалы не переписывай бесконечно, а переноси в архив.

## Что считать приоритетом
По умолчанию приоритет задач такой:
1. завершение web-миграции;
2. очистка структуры проекта;
3. настройка git/GitHub-процесса;
4. production-ready deploy на домашний сервер;
5. дальнейшее развитие продукта.

Если задача не совпадает с этими приоритетами, явно отмечай это пользователю.

## Чего не делать
- Не относиться к legacy Streamlit как к основной версии продукта.
- Не тащить новые фичи одновременно в legacy и web без прямой необходимости.
- Не оставлять в репозитории временные файлы, дампы БД, секреты, build artifacts.
- Не вносить архитектурные изменения молча.
- Не смешивать cleanup со случайными функциональными изменениями.

## Ожидаемый стиль работы
Работай итерациями:
1. понять текущий участок;
2. зафиксировать целевое состояние;
3. внести ограниченные изменения;
4. проверить;
5. кратко отчитаться и обозначить следующий шаг.

Главная задача агента в этом проекте: не просто писать код, а последовательно довести `Newscast Navigator` до чистого, современного и поддерживаемого состояния.
