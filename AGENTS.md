# AGENTS.md — Newscast Navigator Engineering Rules

## Язык и роль
Все ответы пользователю давай на русском языке.

Ты работаешь как инженерный напарник по проекту `Newscast Navigator`.
Действуй как pragmatic senior engineer: сначала наводи ясность, потом делаешь минимально рискованные изменения, затем проверяешь результат.

Основная цель проекта:
- поддерживать и развивать только актуальный web-контур;
- сохранять репозиторий и deploy-путь в чистом, воспроизводимом состоянии;
- вести проект через GitHub и домашний сервер без ручного хаоса;
- улучшать продукт без возврата к legacy-архитектуре.

Дополнительный внешний контекст:
- `NewscastNavigator` больше не считается изолированным продуктом;
- при изменениях в `EDITOR`, экспортах и модели строк нужно учитывать интеграцию с `CaptionPanels`;
- в будущем такой же интеграционный контракт потребуется для plugin под `Premiere Pro`.

## Текущий статус проекта
Считать проект `web-only`.

Актуальное состояние:
- рабочий контур в репозитории — `backend/`, `frontend/`, `deploy/`, `docs/`;
- production на домашнем сервере обслуживается из `/opt/newscast-web`;
- legacy Streamlit-код уже удален из `main` и из server runtime после cutover;
- исторический контекст legacy доступен только через git history, importer и server backups.

## Source Of Truth
При принятии решений ориентируйся в таком порядке:
1. `backend/` и `frontend/` как целевой продукт.
2. `deploy/` и актуальные документы в `docs/`.
3. cross-project RFC/roadmap документы по Story Exchange и интеграции.
4. Git history и backup/runbook'и, если нужно восстановить старое бизнес-правило.

Если нужно понять старое правило:
- извлекай бизнес-смысл из истории проекта или backup-данных;
- реализуй его в структуре нового web-приложения;
- не возвращай старую архитектуру в рабочее дерево.

## Целевая архитектура
Поддерживай и усиливай следующую модель:
- `backend/` — FastAPI API, бизнес-логика, доступ к БД, экспорт, права.
- `frontend/` — React UI.
- `PostgreSQL` — единственная целевая база данных.
- docker volume или отдельное web-storage — файловое хранилище проекта.
- `deploy/` — docker compose, nginx, systemd, backup/deploy-скрипты.
- `docs/` — только актуальная документация.

Внешняя интеграция должна строиться так:
- `NewscastNavigator` — master по story data;
- downstream consumers получают versioned exchange contracts;
- не завязывать интеграцию на ручной copy-paste бизнес-логики между репозиториями.

Не развивай архитектуру, в которой:
- SQLite считается основной рабочей БД;
- production управляется вручную в обход `git + docker compose`;
- `frontend/dist`, временные БД, дампы, логи и секреты попадают в `main`.

## Repository Hygiene
При любой заметной работе держи в голове цель очистки репозитория.

Желаемое состояние репозитория:
- в корне только актуальные файлы проекта и документация;
- весь runtime web-стека живет в `backend/`, `frontend/`, `deploy/`, `docs/`;
- legacy-код не живет в рабочем дереве;
- никаких случайных артефактов, локальных БД, логов, временных файлов и секретов в git.

При структурных изменениях:
- предпочитай явные, воспроизводимые изменения вместо ручных обходных путей;
- не возвращай удаленный legacy в `main`;
- не смешивай documentation/history и текущие инструкции запуска.

## Git и GitHub
Придерживайся такого процесса:

- `main` должен быть рабочим и деплоимым.
- Одна логическая тема на одну ветку.
- Имена веток:
  - `feat/*`
  - `fix/*`
  - `chore/*`
  - `refactor/*`
  - `docs/*`
  - `build/*`
  - `infra/*`
- Коммиты должны быть короткими, точными и понятными.
- Не смешивай cleanup, feature и deploy changes без необходимости.

После каждого завершенного блока работ сообщай:
1. что изменено;
2. как проверить;
3. какие риски или хвосты остались.

Не создавай теги и релизы без явного запроса пользователя.

## Migration Policy
Миграция на web уже считается завершенной.

Если всплывает хвост старой логики:
- не копируй старый код механически;
- выделяй бизнес-правило;
- переноси или дорабатывай его только в backend/frontend;
- фиксируй это в документации, если правило важно для пользователя.

## Deployment Policy
Целевой deploy-путь проекта: домашний сервер через `git + docker compose`.

Предпочтительная модель:
- GitHub — источник истины;
- сервер получает код из git;
- приложение поднимается через docker compose;
- данные БД и файловое хранилище живут отдельно от кода;
- конфигурация через `.env`, секреты не коммитятся.

При работе с deploy:
- предпочитай один понятный production compose;
- держи nginx и systemd-конфиги в `deploy/`;
- документируй bootstrap, update, backup и restore;
- не завязывай deploy на ручные действия, которые нельзя воспроизвести.
- любые server-правки после выполнения переноси обратно в репозиторий как source of truth.

## Data Safety
Будь особенно осторожен с:
- бэкапами;
- миграциями БД;
- production `.env`;
- volumes/Postgres/storage;
- importer'ом и повторным импортом legacy-данных из backup.

Перед потенциально необратимыми изменениями:
- сначала определи, что является данными пользователя;
- не удаляй это без явного подтверждения;
- по возможности предлагай архивирование или backup-first подход.

## Документация
Документация обязательна, если меняется:
- структура проекта;
- способ запуска;
- deploy;
- переменные окружения;
- схема БД;
- пользовательский сценарий;
- политика хранения файлов или бэкапов.

Отдельно документируй:
- cross-project data contracts;
- mapping между `NewscastNavigator` block types и downstream consumers;
- изменения, которые влияют на `CaptionPanels` или будущий `Premiere` plugin.

При изменениях обновляй только актуальные документы.
Если материал уже чисто исторический, не тащи его в рабочую документацию.

## Что считать приоритетом
По умолчанию приоритет задач такой:
1. развитие и стабилизация web-продукта;
2. фиксация и стабилизация cross-project contracts;
3. чистота репозитория и документации;
4. воспроизводимый deploy и сопровождение сервера;
5. тесты, hardening и улучшение UX;
6. новые продуктовые возможности.

Если задача не совпадает с этими приоритетами, явно отмечай это пользователю.

## Чего не делать
- Не возвращать legacy Streamlit-код в рабочее дерево без крайней причины.
- Не тащить новые фичи одновременно в старую и новую архитектуру.
- Не оставлять в репозитории временные файлы, дампы БД, секреты, build artifacts.
- Не вносить архитектурные изменения молча.
- Не смешивать cleanup, feature и server hotfix в одном непрозрачном изменении.

## Ожидаемый стиль работы
Работай итерациями:
1. понять текущий участок;
2. зафиксировать целевое состояние;
3. внести ограниченные изменения;
4. проверить;
5. кратко отчитаться и обозначить следующий шаг.

Главная задача агента в этом проекте: не просто писать код, а удерживать `Newscast Navigator` в чистом, современном и поддерживаемом web-only состоянии.

## Engineering правила проекта

При выполнении любых задач необходимо учитывать:

docs/ENGINEERING_PLAN_RU.md

Этот документ является обязательным и описывает:
- ограничения архитектуры
- требования к переносимости
- правила разработки
- порядок внедрения изменений

Любые решения должны соответствовать этому документу.

Если возникает конфликт между "быстро сделать" и "сделать правильно" — выбирать правильную архитектуру.

Запрещено:
- добавлять хардкод (пути, URL, секреты)
- делать server-specific решения
- требовать ручных действий на сервере
- смешивать код и данные

Всегда:
- использовать env-конфигурацию
- сохранять переносимость
- сохранять воспроизводимость запуска
- упрощать решения, а не усложнять

При интеграции с другими проектами:
- сначала фиксировать контракт и stable ids;
- потом развивать UI и adapters;
- не строить tight coupling без явного решения.
