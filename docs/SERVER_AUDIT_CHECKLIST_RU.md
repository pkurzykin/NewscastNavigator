# Server Audit Checklist

Дата актуализации: 2026-03-15

## Зачем нужен этот документ
На домашнем сервере уже работает какая-то версия `Newscast Navigator`.
Поэтому перед первым deploy нового web-контура нельзя просто делать `docker compose up`.

Сначала нужно понять:
- что именно сейчас запущено;
- где лежат данные;
- какие порты и reverse proxy уже заняты;
- как не сломать существующую рабочую версию.

## Что проверить на сервере

### 1. Где живет приложение
- путь до каталога проекта;
- есть ли там git-репозиторий;
- какая ветка и какой commit сейчас развернуты;
- есть ли локальные незакоммиченные изменения.

### 2. Как оно запускается
- `docker compose`;
- `systemd`;
- вручную через shell;
- несколько параллельных способов сразу.

### 3. Какие контейнеры и сервисы активны
- список контейнеров;
- их имена;
- какие порты проброшены наружу;
- какие volumes подключены.

### 4. Где лежат данные
- база данных;
- project files/storage;
- backup-каталоги;
- старые SQLite-файлы, если они еще используются.

### 5. Что делает reverse proxy
- nginx или другой proxy;
- какие `server_name` уже настроены;
- какие upstream и локальные порты используются;
- где лежат активные конфиги.

### 6. Как устроено восстановление
- есть ли регулярные backup;
- чем восстанавливается база;
- чем восстанавливается storage;
- когда последний раз проверяли restore.

### 7. Что нельзя сломать при cutover
- текущий домен или IP;
- существующий production-порт;
- каталог с данными;
- старые контейнеры, если они пока остаются fallback-вариантом.

## Минимальная команда-инвентаризация
Ниже не “скрипт развертывания”, а безопасный список того, что нужно посмотреть:

```bash
pwd
git status -sb
git rev-parse --short HEAD
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Ports}}\t{{.Status}}'
docker volume ls
systemctl list-units --type=service | grep -Ei 'newscast|docker|nginx'
ss -tulpn | grep -E ':80|:443|:5173|:8000|:8100|:8088|:5432|:5433'
```

Если есть nginx на сервере:

```bash
sudo nginx -T
```

## Готовый вариант для snapshot-съема
Если удобнее не собирать все руками, можно использовать:

```bash
bash deploy/scripts/server_audit_snapshot.sh
```

Скрипт:
- ничего не меняет на сервере;
- складывает снимок состояния в отдельную timestamp-папку;
- помогает потом спокойно разобрать контейнеры, порты, systemd и nginx по файлам.

## Результат аудита должен ответить на 5 вопросов
1. Что именно сейчас считается рабочей production-версией.
2. Где лежат production-данные.
3. На каком порту можно безопасно поднять новый web-контур параллельно.
4. Как откатиться назад, если новый deploy не взлетит.
5. Можно ли делать cutover через существующий reverse proxy или нужен новый.
