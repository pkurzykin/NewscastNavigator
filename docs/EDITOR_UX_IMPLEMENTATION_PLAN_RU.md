# План Улучшений UX/UI Для EDITOR

Дата: 2026-03-26  
Статус: draft for implementation

## 1. Цель

Сделать `EDITOR` быстрее, понятнее и удобнее для ежедневной редакторской работы, не ломая его базовое свойство:

- материал должен восприниматься как единый текстовый поток;
- типы блоков должны считываться мгновенно;
- техполя должны быть всегда видимы и доступны без лишних действий;
- частые действия должны быть быстрыми как мышью, так и с клавиатуры.

## 2. Что сохраняем

При дальнейшей переработке `EDITOR` не превращаем его в набор тяжелых карточек.

Сохраняем:

- табличную основу экрана;
- ощущение единого сценарного полотна сверху вниз;
- всегда видимые техполя;
- интеграционные поля и структуру данных, нужные для `CaptionPanels` и будущего `Premiere` plugin;
- существующую backend-модель строк, rich text, autosave и revision layer.

Не делаем:

- сильные цветные фоны на весь блок;
- жирные рамки вокруг каждого блока;
- скрытие обязательных техполей в аккордеоны и модалки;
- радикальную смену архитектуры `EDITOR`.

## 3. Что уже есть и не требует повторной реализации

В текущем `EDITOR` уже существуют:

- создание строки с выбором типа блока;
- autosave таблицы и workflow;
- авторост высоты большинства текстовых полей;
- rich text formatting;
- несколько `Имя файла / TC` на один блок;
- revision/history слой;
- базовая sticky-toolbar логика.

Поэтому дальше идем не через rebuild с нуля, а через последовательный UX-refactor текущего решения.

## 4. Главные продуктовые принципы

### 4.1. Основной текст главный

Любая строка должна визуально отвечать на вопрос:

- где основной текст;
- что относится к техданным;
- какой это тип блока.

При этом главный акцент остается на тексте.

### 4.2. Поток материала важнее карточности

Все блоки вместе должны ощущаться как один материал, а не как набор независимых виджетов.

### 4.3. Частые действия всегда рядом

Пользователь не должен открывать дополнительные окна или скрытые панели для базовых действий.

### 4.4. Мягкая идентификация вместо агрессивной

Тип блока обозначаем компактным маркером, а не заливкой всей строки.

## 5. Общая дорожная карта

Работу ведем пятью спринтами.

### Sprint 1 — Editor Core Layout

Цель:

- привести строку к понятной анатомии;
- усилить главный текст;
- ослабить техполя по визуальному весу;
- ввести единый save-status;
- нормализовать базовый tab/focus workflow.

### Sprint 2 — Editor Keyboard & Speed

Цель:

- ускорить частые действия;
- ввести keyboard-first workflow;
- сделать редактор ощутимо быстрее в ежедневной работе.

### Sprint 3 — Ordering & Navigation

Цель:

- быстро переставлять блоки;
- добавить outline/navigation для длинных материалов;
- упростить переходы по структуре.

### Sprint 4 — Review Mode

Цель:

- разделить режим редактирования и режим проверки;
- уменьшить визуальный шум при вычитке;
- связать это с уже существующим revision-layer.

### Sprint 5 — Product Polish

Цель:

- унифицировать визуальный язык;
- улучшить empty states и ошибки;
- добавить шаблоны/пресеты для типовых материалов.

## 6. Sprint 1 — конкретный scope

Это следующий основной этап.

### 6.1. Единая анатомия строки

Для каждой строки вводим одну и ту же композицию:

1. `Header line`
- компактный маркер типа блока;
- базовые действия строки;
- минимум визуального шума.

2. `Main content zone`
- основной текст;
- `СНХ`: `ФИО / Должность / Текст`;
- `ЗК+гео`: `Гео / Текст`.

3. `Tech zone`
- `Имя файла / TC`;
- `В кадре`;
- другие служебные поля строки.

Требование:

- пользователь за долю секунды понимает структуру строки;
- текст остается главным визуальным объектом.

### 6.2. Мягкая идентификация типа блока

Нужно:

- заменить тяжелое визуальное разделение на компактный маркер типа;
- использовать короткий label:
  - `ЗК`
  - `ЗК+гео`
  - `СНХ`
  - `ЛАЙФ`
- цвет использовать только в маркере, не в фоне всей строки.

### 6.3. Компактная и постоянная техзона

Техполя всегда видимы.

Нужно:

- сохранить их постоянно открытыми;
- сделать их компактными по высоте;
- привести `Имя файла / TC` к стабильному layout;
- ввести нормализованный формат `TC IN / TC OUT`;
- сделать быстрый tab-order по техполям.

### 6.4. Единый статус сохранения

Нужно:

- один постоянный индикатор сохранения в верхней панели;
- состояния:
  - `Сохранение...`
  - `Сохранено`
  - `Ошибка сохранения`
- показать время последнего успешного сохранения;
- не дублировать это в нескольких местах экрана.

### 6.5. Базовый focus / tab cleanup

Нужно:

- нормализовать порядок перехода `тип -> текст -> техполя -> следующая строка`;
- после создания блока давать предсказуемый фокус;
- убрать случаи, когда пользователь “ловит” курсор мышью.

## 7. Sprint 2 — конкретный scope

### 7.1. Частые действия по строке

Добавить быстрые действия:

- добавить блок;
- удалить блок;
- дублировать блок;
- поднять блок вверх;
- опустить блок вниз.

### 7.2. Горячие клавиши

Минимальный набор:

- добавить блок;
- удалить блок;
- дублировать блок;
- переместить вверх/вниз;
- сохранить вручную;
- перейти к следующему/предыдущему блоку.

### 7.3. Быстрый workflow для выбора типа

Существующую механику `Добавить строку` довести до окончательного варианта:

- понятная кнопка `Добавить блок`;
- быстрый выбор типа;
- предсказуемый фокус после вставки.

## 8. Sprint 3 — конкретный scope

### 8.1. Управление порядком

Сначала:

- кнопки `вверх/вниз`;
- индикатор позиции.

Потом:

- аккуратный drag-and-drop.

### 8.2. Навигация по длинному материалу

Нужно:

- outline по блокам;
- быстрый jump к блоку;
- подсветка текущего блока.

Условие:

- navigation не должна съедать основное пространство текста.

## 9. Sprint 4 — конкретный scope

### 9.1. Два режима представления

`Редактирование`:

- активные input-поля;
- акцент на вводе;
- быстрые действия.

`Проверка`:

- чище чтение;
- меньше визуального шума;
- удобнее видеть правки, комментарии и статусы.

Важно:

- не создавать второй отдельный экран;
- это должны быть два представления одного и того же `EDITOR`.

## 10. Sprint 5 — конкретный scope

### 10.1. Унификация компонентов

Привести к единой системе:

- кнопки;
- поля;
- лейблы;
- отступы;
- иконки;
- статусы;
- действия строки.

### 10.2. Empty states и ошибки

Нужно:

- стартовый экран, если блоков еще нет;
- понятное создание первого блока;
- локальные ошибки у поля, а не хаотичные глобальные сообщения;
- прикладная валидация `TC`.

### 10.3. Шаблоны материалов

Добавить типовые пресеты структуры только после стабилизации базового workflow.

## 11. Практический порядок внедрения

Идем малыми шагами.

### Step A

Сначала делаем `Sprint 1 / 6.1 + 6.2`:

- анатомия строки;
- мягкий маркер типа блока.

### Step B

Потом `Sprint 1 / 6.3`:

- окончательно стабилизируем техзону.

### Step C

Потом `Sprint 1 / 6.4 + 6.5`:

- единый save-status;
- tab/focus cleanup.

Только после этого переходим к ускорению workflow и навигации.

## 12. Что сознательно не делаем в первом спринте

Не делаем сразу:

- drag-and-drop;
- отдельный review mode;
- шаблоны материалов;
- большой redesign revision-layer;
- новые backend-фичи без крайней необходимости.

## 13. Риски

Главный риск:

- случайно усилить карточность и разрушить ощущение текстового потока.

Второй риск:

- смешать UX-refactor с новыми продуктовыми фичами и потерять фокус.

Третий риск:

- начать менять backend там, где достаточно frontend/layout refactor.

## 14. Definition of Done для Sprint 1

Sprint 1 считаем завершенным, если:

- строка визуально читается как часть единого материала;
- тип блока определяется мгновенно;
- техполя всегда видимы и не доминируют;
- статус сохранения понятен без чтения мелких подсказок;
- tab/focus workflow не мешает набору материала;
- пользователь может работать в `EDITOR` весь день без ощущения “набора разрозненных полей”.
