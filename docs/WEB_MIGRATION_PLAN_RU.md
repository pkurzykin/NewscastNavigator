## 1. Зачем переходить со Streamlit
Текущее решение на Streamlit хорошо сработало как быстрый MVP, но дальше появляются ограничения:
- сложно сделать UI точно как в рабочих табличных референсах;
- сложно держать большой интерфейс и сложные сценарии в одном `app.py`;
- ограниченный контроль над интерактивностью таблиц и навигацией;
- сложнее масштабировать и поддерживать долгосрочно.

Вывод: для следующего этапа развития целесообразно переходить на классическую архитектуру:
- Backend API
- Frontend SPA
- отдельная БД

## 2. Рекомендуемый стек (актуально и эффективно)

## Backend
- `FastAPI` — быстрый, современный Python backend, удобная валидация и Swagger.
- `SQLAlchemy 2.x` — ORM/SQL-слой.
- `Alembic` — миграции БД.
- `Pydantic v2` — схемы запросов/ответов.
- `Uvicorn` (dev) + `Gunicorn + Uvicorn workers` (prod).

Почему: это стандартный и актуальный стек для Python API-проектов в 2026.

## База данных
- `PostgreSQL 16/17`.

Почему:
- надежнее и удобнее для многопользовательской работы, чем SQLite;
- лучше конкуренция транзакций;
- проще рост проекта (индексы, сложные запросы, аудит).

## Frontend
- `React 18 + TypeScript + Vite`.
- `React Router` — маршруты (`/main`, `/editor/:id`, `/archive`).
- `TanStack Query` — работа с API, кеш, загрузки, обновления.
- `AG Grid Community` — редактируемая таблица уровня "Excel-like" с выделением строк.
- `MUI` или `Mantine` — базовые UI-компоненты.

Почему:
- максимальный контроль UI/UX;
- удобная реализация табличного редактора и ролей;
- простой путь для долгого развития.

## Аутентификация и доступ
- Cookie-based сессия (HTTPOnly cookie + таблица сессий в БД).
- RBAC по ролям (`admin`, `editor`, `author`, `proofreader`).

Почему:
- проще и безопаснее для внутреннего веб-приложения;
- легко отзывать сессии и контролировать доступ.

## Файлы
- Хранение только на сервере (как вы и требуете):
  - `/srv/newscast-navigator/storage/projects/<project_id>/...`
- В БД хранить метаданные/пути, а не бинарные файлы.

## Экспорты
- `python-docx` для DOCX.
- `reportlab` или `WeasyPrint` для PDF (на старте можно оставить `reportlab`, чтобы не ломать текущую логику).

## Тесты
- Backend: `pytest`, `pytest-asyncio`, `httpx`.
- Frontend: `vitest` + `playwright` для e2e критичных сценариев.

## Инфраструктура
- `Docker Compose` для dev/prod (разные compose-файлы или профили).
- `Nginx` как reverse proxy.
- `systemd` для автозапуска Docker/Compose.

## 4. Как сохранить текущий функционал без потерь

Подход: "strangler migration" (поэтапный перенос).

1. Стабилизируем текущий Streamlit MVP (как fallback).
2. Выносим бизнес-логику из `app.py` в отдельные сервисы/репозитории.
3. Поднимаем backend API, который повторяет текущие операции:
   - auth/roles
   - проекты (main/archive)
   - элементы сценария (editor)
   - комментарии проекта
   - файлы проекта
   - экспорт
4. Делаем frontend-экраны в том же порядке: `MAIN -> EDITOR -> ARCHIVE`.
5. Сверяем паритет по чек-листу.
6. Переключаем пользователей на новый web.
7. Streamlit оставляем только как резерв и потом удаляем.

## 5. Паритет-функционал (обязательный список)
- Логин/роли/ограничения редактирования.
- MAIN:
  - список проектов;
  - создание нового/на основе выбранного/последнего;
  - фильтры;
  - отправка в архив.
- EDITOR:
  - единая редактируемая таблица;
  - добавление/удаление строк;
  - валидации TC и СНХ (2 строки в титре);
  - комментарии проекта;
  - файлы проекта.
- ARCHIVE:
  - список архивных;
  - возврат в работу.
- Экспорт DOCX/PDF.
- Печатная версия.

## 7. Риски и как их снять
- Риск: потеря паритета функций.  
  Решение: фиксируем чек-лист паритета и принимаем релиз только после полного совпадения.

- Риск: миграция БД и данных.  
  Решение: сначала перенос схемы, потом пробный импорт, потом контрольная сверка.

- Риск: усложнение для первого запуска.  
  Решение: идем строго спринтами, не расширяем scope до завершения паритета.

## 8. Практические рекомендации "что где использовать"
- Таблица редактора (`EDITOR`): `AG Grid` (frontend), backend bulk API.
- Роли/доступ: только на backend (frontend лишь скрывает кнопки, но не решает безопасность).
- Валидации данных: дублировать и на frontend, и на backend.
- Файлы: локальная ФС + БД-метаданные.
- Бэкапы: nightly backup PostgreSQL + storage каталога.
- Dev/Prod: отдельные compose-файлы, не смешивать.
