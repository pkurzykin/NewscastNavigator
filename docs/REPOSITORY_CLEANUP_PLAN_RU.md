# План cleanup репозитория

Дата актуализации: 2026-03-15

## Зачем нужен этот документ

Сейчас в проекте одновременно живут:
- legacy Streamlit-версия;
- новая web-версия (`backend + frontend`);
- локальные dev-артефакты;
- старые deploy-файлы.

Из-за этого репозиторий выглядит перегруженным. Этот документ фиксирует безопасный порядок cleanup без риска потерять нужную логику.

## Целевое состояние

В `main` должно остаться только то, что относится к актуальной web-версии:
- `backend/`
- `frontend/`
- `deploy/`
- `docs/`

Legacy должен быть либо:
- перенесен в отдельную папку `legacy/streamlit_mvp/`, либо
- вынесен в отдельную ветку/архив до удаления из основной ветки.

## Что уже сделано

- Добавлен корневой `README.md`.
- Добавлен `.gitignore` под будущий GitHub-репозиторий.
- Уточнены `backend/README.md` и `frontend/README.md`.
- Обновлен deployment-документ с пометкой, что сервер нельзя трогать без аудита.
- Явный legacy-контур физически вынесен в `legacy/streamlit_mvp/`.

## Следующие шаги cleanup

### Этап 1. Дочистить активную web-структуру

Нужно:
- оставить актуальные инструкции запуска только для web-контура;
- убрать двусмысленность в deploy-документации;
- проверить, какие файлы в `deploy/` реально относятся к новому web, а какие еще legacy.

Критерий завершения:
- любой новый разработчик может быстро понять, что `backend/` и `frontend/` — это основное приложение.

### Этап 2. Отделить legacy явно

Статус: выполнено частично.

Что уже сделано:
- старые Streamlit-файлы вынесены из корня;
- legacy deploy-артефакты вынесены из `deploy/`;
- старая SQLite-структура и локальные legacy-данные собраны в одной зоне.

Что еще важно довести:
- убрать оставшиеся двусмысленные ссылки на старые пути из документации;
- решить судьбу пустых/служебных корневых директорий и локального мусора перед первым git-оформлением.

Критерий завершения:
- в корне проекта не остается смешения двух архитектур.

### Этап 3. Удалить локальный мусор перед первым git-оформлением

Нужно:
- не включать в будущий git локальные `venv`, `node_modules`, `dist`, `__pycache__`, временные БД, backup и runtime-папки;
- при необходимости физически удалить их из рабочего дерева перед первой инициализацией репозитория.

Критерий завершения:
- рабочая папка визуально чистая и готова к первому осмысленному коммиту.

### Этап 4. Подготовить git/GitHub

Нужно:
- инициализировать git;
- создать первый чистый коммит уже после cleanup;
- завести GitHub-репозиторий;
- зафиксировать правила веток и коммитов.

Критерий завершения:
- `main` отражает только актуальное состояние проекта и не тянет за собой локальный шум.

## Что нельзя делать в cleanup

- Нельзя удалять legacy раньше, чем новый web закроет нужный паритет.
- Нельзя трогать домашний сервер до отдельного аудита.
- Нельзя смешивать cleanup репозитория с production deploy.

## Практический вывод

Сейчас cleanup нужно вести не через массовые удаления, а поэтапно:
1. сначала сделать репозиторий понятным;
2. потом отделить legacy;
3. потом оформить git/GitHub;
4. только после этого идти к серверу и production.
