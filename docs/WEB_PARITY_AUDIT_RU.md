# Web Parity Audit — Legacy Streamlit vs New Web

Дата: 2026-03-15

## Зачем нужен этот документ
Это рабочая карта завершения миграции `Newscast Navigator` со старого `Streamlit + SQLite` на новый `FastAPI + React + PostgreSQL`.

Документ отвечает на три вопроса:
- что уже перенесено в новый web-контур;
- что перенесено частично;
- что еще нужно сделать до безопасной очистки репозитория от legacy-кода.

## Короткий вывод
Новый web-контур уже покрывает основной цикл работы:
- логин;
- список проектов `MAIN/ARCHIVE`;
- создание и клонирование проектов;
- сценарная таблица `EDITOR`;
- проектные комментарии;
- проектные файлы;
- экспорт `DOCX/PDF`.

Но migration еще не завершен.
В legacy остаются важные workflow-возможности, которые в новом web либо отсутствуют, либо упрощены:
- полноценные статусы проекта;
- назначение ролей по проекту;
- история событий проекта;
- более богатые фильтры `MAIN/ARCHIVE`;
- печатная версия;
- часть правил доступа, связанных с ролью `proofreader`.

Вывод:
- новый web уже можно считать основной платформой развития;
- legacy еще нельзя бездумно удалить;
- перед cleanup нужно закрыть список разрывов ниже.

Примечание по состоянию кода на 2026-03-15:
- в backend уже начата реализация workflow-foundation для паритета с legacy;
- добавлены поля workflow/архива и таблица `project_events`;
- добавлены backend-эндпоинты для истории проекта и расширенной мета-модели;
- frontend и end-to-end сценарии еще не доведены до нового API, поэтому user-facing паритет по этому документу пока считать закрытым нельзя.

## Статусы оценки
- `DONE` — функция по сути перенесена.
- `PARTIAL` — работает, но урезана по сравнению с legacy.
- `MISSING` — в новом web отсутствует.
- `DECIDE` — нужно отдельно решить, нужна ли функция в целевой версии.

## Что уже можно считать успешно перенесенным
Эти функции больше не держат проект на legacy:
- базовый логин;
- основной список проектов;
- создание и клонирование проектов;
- архивирование и возврат;
- таблица сценария;
- валидации `TC` и `СНХ`;
- проектные комментарии;
- проектные файлы;
- экспорт `DOCX/PDF`.

Если смотреть только на базовый пользовательский цикл "создал -> отредактировал -> приложил файлы -> экспортировал", новый web уже жизнеспособен.

## Критические разрывы перед cleanup legacy
Перед тем как удалять или глубоко архивировать legacy-код, нужно закрыть минимум:

1. `Workflow/статусы`.
Сейчас новый web слишком сильно упростил модель проекта.
В legacy были статусы:
- `draft`
- `reviewed`
- `in_editing`
- `in_proofreading`
- `ready`
- `delivered`
- `archived`

В новом web фактически живет только сценарий `draft <-> archived`.

2. `Назначения по проекту`.
В legacy были отдельные поля:
- автор;
- исполнитель;
- корректор.

В новом web есть только `author_user_id`.

3. `История событий`.
В legacy есть `project_events`, и это полезно не только для аудита, но и для восстановления из архива.
В новом web этой модели нет совсем.

4. `Архивные метаданные`.
В legacy есть кто и когда архивировал проект.
В новом web это не хранится, поэтому архив уже сейчас менее управляемый.

5. `Фильтры MAIN/ARCHIVE`.
В web пока только базовый поиск.
Для реальной редакционной работы этого почти наверняка мало.

## Что можно не считать блокером для cleanup
Эти вещи полезны, но не обязаны останавливать чистку репозитория:
- титровальный режим;
- быстрый экспорт из `MAIN`;
- подтверждение действия через отдельный шаг;
- AG Grid вместо простой таблицы;
- дизайнерская роль, если она больше не нужна;
- печатная версия, если экспорт `PDF` покрывает рабочую задачу.

## Рекомендуемый порядок завершения миграции

### Этап 1. Закрыть модель данных и workflow
Сначала вернуть в новый backend сущности, которые определяют поведение системы:
- расширенный список статусов;
- поля `executor_user_id`, `proofreader_user_id`;
- поля архивации;
- `source_project_id`;
- `project_events`.

Почему сначала это:
- без этого нельзя честно восстановить бизнес-логику;
- иначе потом придется переделывать API и UI повторно.

### Этап 2. Дотянуть backend API
После расширения модели:
- API обновления метаданных проекта;
- API смены статуса проекта;
- API истории проекта;
- API фильтров `MAIN/ARCHIVE`;
- корректное восстановление из архива в предыдущий статус.

### Этап 3. Дотянуть frontend до рабочего паритета
Потом уже на frontend:
- форма метаданных проекта;
- управление статусом;
- назначение автора/исполнителя/корректора;
- история проекта в `ARCHIVE` или `EDITOR`;
- расширенные фильтры списков.

### Этап 4. Cleanup legacy
Когда пункты выше закрыты:
- зафиксировать legacy в отдельной ветке или архивной папке;
- удалить старый runtime из `main`;
- убрать старые инструкции запуска;
- оставить только web-контур.

### Этап 5. Post-migration hardening
После cleanup:
- нормальный production compose;
- git/GitHub workflow;
- CI smoke checks;
- backup/restore;
- затем уже UI-polish и новые фичи.

## Решения, которые надо принять отдельно
Перед реализацией полезно принять несколько простых решений:

1. Нужна ли в новом проекте роль `designer`.
2. Нужна ли печатная версия, если `PDF`-экспорт достаточен.
3. Нужен ли быстрый экспорт прямо из `MAIN`.
4. Нужно ли вводить полноценные cookie-сессии сейчас или оставить bearer token до этапа hardening.
5. Остается ли `proofreader` обязательной рабочей ролью.

## Практический вывод
Если цель: "завершить переезд и начать спокойно развивать проект", то правильный следующий фокус не cleanup, а закрытие 5 разрывов:
- статусы;
- назначения;
- история событий;
- архивные метаданные;
- фильтры.

После этого репозиторий уже можно будет чистить без страха потерять скрытую рабочую логику, которая сейчас еще живет только в legacy Streamlit.
