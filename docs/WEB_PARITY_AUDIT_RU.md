# Web Parity Audit — Legacy Streamlit vs New Web

Дата: 2026-03-15

## Зачем нужен этот документ
Это рабочая карта завершения миграции `Newscast Navigator` со старого `Streamlit + SQLite` на новый `FastAPI + React + PostgreSQL`.

Документ отвечает на три вопроса:
- что уже перенесено в новый web-контур;
- что перенесено частично;
- что еще нужно сделать до безопасной очистки репозитория от legacy-кода.

## Короткий вывод
Новый web-контур уже покрывает основной цикл работы:
- логин;
- список проектов `MAIN/ARCHIVE`;
- создание и клонирование проектов;
- сценарная таблица `EDITOR`;
- workflow-метаданные проекта;
- назначения автора/исполнителя/корректора;
- история проекта;
- расширенные фильтры `MAIN/ARCHIVE`;
- проектные комментарии;
- проектные файлы;
- экспорт `DOCX/PDF`.

Но migration еще не завершен.
В legacy и новом web еще нужно добить несколько завершающих вещей:
- полный ручной `e2e`-прогон нового контура;
- финальные решения по печатной версии и быстрому экспорту из `MAIN`;
- hardening auth/deploy перед production;
- точечная проверка прав доступа по ролям в реальном запуске.

Вывод:
- новый web уже можно считать основной платформой развития;
- зависимость от legacy по ключевой бизнес-логике уже в основном снята;
- дальше фокус должен смещаться с миграции модели на проверку, hardening и deploy.

Примечание по состоянию кода на 2026-03-15:
- backend и frontend уже переведены на расширенную workflow-модель;
- в web-контуре есть статусы проекта, назначения, архивные метаданные, история и расширенные фильтры;
- `EDITOR` и `WORKSPACE` теперь используют ту же богатую мета-модель проекта, что и `MAIN`;
- правила редактирования для `proofreader` и статуса `in_proofreading` приведены ближе к legacy-логике;
- главным оставшимся риском считается не нехватка модели, а нехватка полноценной runtime-проверки и production-hardening.

## Статусы оценки
- `DONE` — функция по сути перенесена.
- `PARTIAL` — работает, но урезана по сравнению с legacy.
- `MISSING` — в новом web отсутствует.
- `DECIDE` — нужно отдельно решить, нужна ли функция в целевой версии.

## Что уже можно считать успешно перенесенным
Эти функции больше не держат проект на legacy:
- базовый логин;
- основной список проектов;
- создание и клонирование проектов;
- архивирование и возврат;
- статусы и назначения по проекту;
- история событий проекта;
- расширенные фильтры `MAIN/ARCHIVE`;
- таблица сценария;
- валидации `TC` и `СНХ`;
- проектные комментарии;
- проектные файлы;
- экспорт `DOCX/PDF`.

Если смотреть только на базовый пользовательский цикл "создал -> отредактировал -> приложил файлы -> экспортировал", новый web уже жизнеспособен.

## Критические разрывы перед финальным отказом от legacy
Перед тем как считать migration полностью закрытой, нужно добить минимум:

1. `E2E-проверка нового web-контура`.
Сейчас модель и UI уже почти на месте, но нужен честный прогон сценариев:
- логин;
- `MAIN`;
- `ARCHIVE`;
- `EDITOR`;
- `WORKSPACE`;
- экспорт;
- права `author/editor/proofreader/admin`.

2. `Production-ready deploy`.
Нужно довести production compose, `.env`, storage policy, backup/restore и безопасную схему обновления на домашнем сервере.

3. `Финальные продуктовые решения`.
Надо отдельно решить судьбу:
- печатной версии;
- быстрого экспорта из `MAIN`;
- роли `designer`;
- возможного перехода с bearer token на cookie/session модель.

## Что можно не считать блокером для cleanup
Эти вещи полезны, но не обязаны останавливать чистку репозитория:
- титровальный режим;
- быстрый экспорт из `MAIN`;
- подтверждение действия через отдельный шаг;
- AG Grid вместо простой таблицы;
- дизайнерская роль, если она больше не нужна;
- печатная версия, если экспорт `PDF` покрывает рабочую задачу.

## Рекомендуемый порядок завершения migration

### Этап 1. Runtime-проверка нового web-контура
Сначала подтвердить, что текущая web-версия действительно проходит руками основной цикл:
- `MAIN`;
- `ARCHIVE`;
- `EDITOR`;
- `WORKSPACE`;
- экспорт;
- права по ролям.

### Этап 2. Cleanup хвостов migration
После проверки:
- обновить документацию под фактическое состояние;
- убрать оставшиеся дубли и мертвые участки внутри web-контура;
- решить, что точно не переносится из legacy.

### Этап 3. Production/deploy foundation
Потом:
- production compose;
- `.env.example`;
- nginx/systemd;
- policy по volumes и storage;
- backup/restore;
- audit домашнего сервера перед первым аккуратным обновлением.

### Этап 4. Hardening
После deploy-подготовки:
- CI smoke checks;
- auth/session hardening;
- runtime logging;
- затем уже UI-polish и новые фичи.

## Решения, которые надо принять отдельно
Перед реализацией полезно принять несколько простых решений:

1. Нужна ли в новом проекте роль `designer`.
2. Нужна ли печатная версия, если `PDF`-экспорт достаточен.
3. Нужен ли быстрый экспорт прямо из `MAIN`.
4. Нужно ли вводить полноценные cookie-сессии сейчас или оставить bearer token до этапа hardening.
5. Остается ли `proofreader` обязательной рабочей ролью.

## Практический вывод
Если цель: "завершить переезд и начать спокойно развивать проект", то migration по модели уже почти закрыта. Следующий правильный фокус:
- реальный запуск и ручная проверка web-контура;
- production-подготовка и аудит домашнего сервера;
- затем hardening и только после этого новые крупные фичи.

То есть проект уже находится не в фазе "перенести основу", а в фазе "доказать, что новый web можно безопасно считать единственной рабочей версией".
