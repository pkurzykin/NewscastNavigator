# Web Smoke Checklist

Дата актуализации: 2026-03-15

## Зачем нужен этот документ
Это короткий ручной чек-лист для подтверждения, что новый web-контур реально работает как основная версия проекта.

Использовать его:
- локально после заметных изменений в `backend/` или `frontend/`;
- перед первым аккуратным обновлением домашнего сервера;
- после изменений в правах, workflow, архиве, экспорте или работе с файлами.

Важно:
- сначала прогоняем локально;
- сервер не трогаем, пока не проведен отдельный аудит того, что уже работает на домашнем сервере.

## Минимальные предусловия
- backend поднят;
- frontend поднят;
- доступна тестовая БД;
- включен demo seed или есть тестовые данные.

Demo-пользователи по умолчанию:
- `admin / admin123`
- `editor / editor123`
- `author / author123`
- `proofreader / proof123`

## Чек-лист

### 1. Логин и базовый запуск
- Зайти под `admin`.
- Убедиться, что открывается `MAIN`, список проектов загружается, ошибок API нет.
- Перелогиниться под `editor`, `author`, `proofreader` и убедиться, что вход работает для всех ролей.

### 2. MAIN
- В `MAIN` загрузить список рабочих проектов.
- Проверить фильтры:
  - поиск по названию;
  - статусы;
  - рубрика;
  - участник;
  - дата создания.
- Создать пустой проект.
- Создать копию последнего проекта.
- Создать копию выбранного проекта.
- Открыть созданный проект в `EDITOR`.

### 3. ARCHIVE
- Отправить проект в архив под `admin` или `editor`.
- Переключиться в `ARCHIVE`.
- Проверить фильтры:
  - название;
  - рубрика;
  - кто архивировал;
  - дата архивации.
- Восстановить проект и убедиться, что он возвращается в предыдущий рабочий статус, а не просто в `draft`.

### 4. EDITOR: таблица и workflow
- Открыть рабочий проект.
- Проверить, что в шапке и блоке метаданных видны:
  - статус;
  - источник клона;
  - время последнего изменения статуса;
  - автор;
  - исполнитель;
  - корректор;
  - архивные метаданные для архивного проекта.
- Изменить:
  - название;
  - рубрику;
  - хронометраж;
  - статус;
  - назначения.
- Добавить, удалить и сохранить строки сценария.
- Проверить валидации:
  - неверный `TC IN/TC OUT`;
  - `TC OUT < TC IN`;
  - у `СНХ` заполнены не все обязательные части: `ФИО` и `Должность`.

### 5. Права по ролям
- Под `author`:
  - в статусах кроме `archived` и `in_proofreading` строки должны редактироваться;
  - в `in_proofreading` строки и workspace должны быть заблокированы;
  - архивный проект должен быть полностью read-only.
- Под `proofreader`:
  - в `in_proofreading` строки и workspace должны редактироваться;
  - смена статуса должна быть доступна;
  - вне архива проект не должен быть read-only.
- Под `editor` и `admin`:
  - редактирование должно работать во всех рабочих статусах;
  - архив должен оставаться read-only до восстановления.

### 6. WORKSPACE
- Проверить сохранение:
  - `file_root`;
  - `project_note`.
- Добавить комментарий.
- Удалить свой комментарий.
- Загрузить файл допустимого типа.
- Скачать файл.
- Удалить файл.
- Убедиться, что после загрузки файла в истории появляется событие `file_uploaded`.

### 7. История проекта
- Открыть блок истории.
- Убедиться, что видны минимум такие события:
  - `project_created`;
  - `project_cloned`, если проект был создан копированием;
  - `status_changed`;
  - `project_archived`;
  - `project_restored`;
  - `file_uploaded`, если загружался файл.

### 8. Экспорт
- Сформировать `DOCX`.
- Сформировать `PDF`.
- Убедиться, что в экспорте корректно подтягиваются:
  - название;
  - рубрика;
  - автор;
  - исполнитель;
  - корректор;
  - строки сценария.

## Что считать стоп-фактором
- backend или frontend не поднимается;
- `MAIN` или `ARCHIVE` не загружаются;
- `EDITOR` получает неполные метаданные проекта;
- роли дают доступ не тем пользователям;
- архивный проект редактируется;
- restore не возвращает проект в предыдущий статус;
- экспорт не формируется;
- файлы загружаются, но история проекта не пополняется.

## Что делать после smoke-check
- если локально все прошло, можно переходить к production/deploy-подготовке;
- если найдены баги, сначала закрыть их локально;
- перед работой с домашним сервером отдельно зафиксировать:
  - как там сейчас запущен проект;
  - где лежат данные;
  - какие контейнеры, volumes, compose-файлы и reverse proxy уже используются.
