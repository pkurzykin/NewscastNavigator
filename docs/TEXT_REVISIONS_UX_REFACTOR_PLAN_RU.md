# TEXT REVISIONS UX REFACTOR PLAN

## UX-переосмысление истории версий в `EDITOR`

Дата: 2026-03-26
Статус: planned

Статус на 2026-03-26:

- `Phase UX-1` реализован: история версий вынесена в right-side panel, базовая терминология упрощена, быстрые действия подняты в sticky toolbar, branch/merge спрятаны в `Дополнительно`.
- `Phase UX-2` реализован: diff стал более человекочитаемым, summary получил цветовые плашки, изменения шапки и строк показываются через `Было / Стало`, а строки сгруппированы по типу изменения.
- `Phase UX-3` и далее остаются в работе как следующие UX-итерации.

---

## 1. Зачем нужен отдельный UX-этап

`TEXT REVISIONS RFC` уже реализован функционально:

- есть immutable snapshot revisions;
- есть diff;
- есть workflow `draft -> submitted -> approved/rejected`;
- есть branch/merge;
- всё это работает на backend и уже выведено в `EDITOR`.

Проблема в том, что текущая реализация получилась **backend-first**:

- пользователь видит слишком много технических сущностей;
- основной сценарий работы с версиями не читается интуитивно;
- важные действия смешаны с редкими и продвинутыми;
- блок версий выглядит как ещё одна техническая карточка, а не как знакомая история документа;
- diff ближе к debug-view, чем к редакторскому сравнению версий.

Следующий этап должен изменить не модель данных, а **способ взаимодействия пользователя с уже существующей системой**.

---

## 2. Цель

Сделать UX истории версий ближе к сочетанию:

- `Google Sheets` для самой таблицы сценария;
- `Google Docs / Word Version History` для работы с версиями.

То есть:

- `EDITOR` остаётся рабочей таблицей;
- история версий становится вторичным интерфейсом;
- обычный пользователь видит только простой сценарий:
  - сохранить версию;
  - отправить на согласование;
  - открыть историю;
  - сравнить;
  - восстановить;
  - сделать текущей.

Технические детали branch/merge и внутренние поля revision-layer должны быть скрыты или показаны только в расширенном режиме.

---

## 3. Ограничения и инварианты

Этот UX-этап **не должен**:

- переписывать backend-модель `project_revisions`;
- ломать текущий API revision-layer;
- ломать already-working workflow в `CaptionPanels` и других downstream-контрактах;
- переводить систему на per-character track changes как в Word.

Этот UX-этап **должен**:

- переиспользовать уже существующие backend API;
- остаться совместимым с текущими ролями и правами;
- не менять архитектуру проекта;
- не добавлять hardcoded сценарии или server-specific решения.

---

## 4. UX-принципы

### 4.1 История версий — вторичный слой

История версий не должна жить как крупная нижняя карточка в основном редакторе.  
Основной экран должен оставаться про написание и редактирование текста.

### 4.2 Один главный сценарий на роль

Для автора:

- редактировать;
- сохранить версию;
- отправить на согласование.

Для редактора:

- открыть историю версий;
- сравнить версии;
- утвердить / отклонить;
- сделать текущей;
- при необходимости восстановить как рабочую.

### 4.3 Прогрессивное раскрытие сложности

Редкие действия не должны конкурировать с основными.

По умолчанию пользователь не должен видеть как главные действия:

- `branch_key`;
- `revision_kind`;
- branch creation;
- merge to main.

Эти действия допустимо оставить только в расширенном режиме и только для нужных ролей.

### 4.4 Человеческая терминология

UI должен говорить языком пользователя, а не языка backend-модели.

### 4.5 Diff должен быть редакторским, а не техническим

Сравнение должно объяснять:

- что добавлено;
- что удалено;
- что изменено;
- что перемещено;

а не просто перечислять поля и внутренние идентификаторы.

---

## 5. Новая терминология

### Что переименовать в UI

| Текущее | Целевое |
| --- | --- |
| `Версии текста` | `История версий` |
| `Создать версию` | `Сохранить версию` |
| `Restore to workspace` / `Восстановить` | `Открыть как рабочую` |
| `Make current` / `Сделать текущей` | `Сделать текущей` |
| `draft` | `Черновик` |
| `submitted` | `На согласовании` |
| `approved` | `Утверждено` |
| `rejected` | `Отклонено` |

### Что скрыть из основного UI

Не показывать в главном пользовательском потоке:

- `branch_key`;
- `revision_kind`;
- raw `segment_uid`;
- `merge` как обычную видимую кнопку.

---

## 6. Целевой информационный дизайн

### 6.1 Основной экран `EDITOR`

В верхней sticky-панели оставить пользовательские действия:

- `Сохранить таблицу`
- `Сохранить версию`
- `Отправить на согласование`
- `История версий`

Принцип:

- основные действия всегда наверху;
- история версий открывается отдельно и не перегружает сам экран таблицы.

### 6.2 История версий как side panel / modal

Вместо нижней карточки использовать один из двух вариантов:

1. правый side panel;
2. большое modal-окно.

Предпочтительный вариант: **правый side panel**.

Почему:

- не уводит пользователя со страницы редактора;
- сохраняет ощущение, что он работает с тем же документом;
- естественно повторяет паттерн version history из docs-like интерфейсов.

### 6.3 Структура side panel

Панель делится на 3 зоны:

1. Верхняя зона:
   - заголовок `История версий`
   - кнопка закрытия
   - фильтр или быстрый переключатель статусов при необходимости

2. Левая/основная зона списка:
   - список версий:
     - номер
     - название
     - дата/время
     - автор
     - статус
     - метка `Текущая`

3. Зона деталей выбранной версии:
   - комментарий
   - шапка (`Название`, `Рубрика`, `Хронометраж`)
   - preview строк
   - действия
   - переключение в режим сравнения

## 7. Новый пользовательский flow

### 7.1 Автор / сценарист

Целевой flow:

1. Пользователь редактирует таблицу.
2. Нажимает `Сохранить версию`.
3. При необходимости добавляет краткий комментарий.
4. Версия появляется в `Истории версий` как `Черновик`.
5. Нажимает `Отправить на согласование`.

Автору не нужно видеть:

- branch;
- merge;
- internal status logic;
- технические поля сравнения.

### 7.2 Редактор

Целевой flow:

1. Открывает `Историю версий`.
2. Выбирает версию.
3. Нажимает `Сравнить`.
4. Смотрит понятный diff.
5. Нажимает:
   - `Утвердить`, или
   - `Отклонить`.
6. При необходимости нажимает `Сделать текущей`.
7. Если нужно вернуть старую версию в работу, нажимает `Открыть как рабочую`.

### 7.3 Продвинутый редакционный режим

Branch/merge оставить, но вынести:

- в `Дополнительно`;
- или в отдельный раздел `Расширенные действия`;
- или показывать только для `editor/admin`.

По умолчанию пользователь не должен видеть это первым экраном.

---

## 8. Целевой дизайн diff

### 8.1 Верхний summary

Сверху показывать сводку:

- `+N добавлено`
- `N удалено`
- `N изменено`
- `N перемещено`

### 8.2 Diff шапки

Шапку сравнивать отдельно:

- `Название`
- `Рубрика`
- `Хронометраж`

И показывать как обычные человекочитаемые изменения:

- было;
- стало.

### 8.3 Diff строк

Каждую строку diff показывать как человеческий блок:

- номер/позиция;
- тип блока;
- короткий фрагмент текста;
- бейдж изменения.

Не показывать raw `segment_uid` как главный ориентир.

### 8.4 Цветовая логика

- зелёный: добавлено
- красный: удалено
- жёлтый: изменено
- синий: перемещено

### 8.5 Уровень детализации

На этом этапе diff остаётся **snapshot-level**, а не inline word-by-word track changes.

Это важно сохранить как осознанное ограничение.

## 9. Поэтапный план внедрения

### Phase UX-1. IA и терминология

Цель:

- убрать нижнюю revision-card из основного потока;
- перенести историю версий в side panel;
- заменить технические названия на пользовательские.

Изменения:

- новая кнопка `История версий` в верхней панели;
- новая кнопка `Сохранить версию`;
- скрытие `branch_key` и `revision_kind` из главного списка;
- role-aware primary actions.

Результат:

- интерфейс перестаёт выглядеть как debug-console.

Статус:

- реализовано

### Phase UX-2. Новый diff view

Цель:

- превратить текущий diff preview в понятное сравнение версий.

Изменения:

- summary-блок;
- шапка изменений;
- список строк с нормальными бейджами;
- скрытие внутренних технических деталей.

Результат:

- редактор понимает, что изменилось, без чтения внутренней модели.

Статус:

- реализовано

### Phase UX-3. Прогрессивное скрытие advanced actions

Цель:

- оставить branch/merge доступными, но не мешающими обычному сценарию.

Изменения:

- advanced-dropdown / advanced-section;
- показывать branch/merge только там;
- при необходимости привязать видимость к ролям.

Результат:

- обычный пользователь не сталкивается с лишней сложностью.

### Phase UX-4. Дошлифовка состояний

Цель:

- сделать поведение панели устойчивым и понятным.

Изменения:

- empty states;
- loading states;
- явные success/error сообщения;
- предсказуемое закрытие/открытие панели;
- сохранение выбранной версии при перезагрузке данных, если это уместно.

Результат:

- интерфейс начинает ощущаться продуктовым, а не инженерным.

---

## 10. Что можно переиспользовать без backend-переделки

Текущая backend-основа уже достаточна для этого UX-этапа:

- list/get/elements;
- create revision;
- diff;
- submit/approve/reject;
- restore;
- mark current;
- branch/merge.

Это означает, что следующий этап почти полностью должен быть **frontend/UX refactor**, а не новой серверной разработкой.

Допустимы только точечные backend-изменения, если при UI-рефакторе выяснится, что нужен более удобный shape ответа.

---

## 11. Что не входит в этот этап

Не входит:

- realtime collaborative editing;
- inline track changes по словам/символам;
- принятие отдельных микроправок внутри одной строки;
- полная модель ревизий как в Google Docs internals;
- переписывание revision-layer заново.

---

## 12. Ручная проверка после внедрения UX-этапа

### Автор

Проверить:

1. редактирование таблицы;
2. `Сохранить версию`;
3. появление версии в истории;
4. `Отправить на согласование`.

### Редактор

Проверить:

1. открытие `Истории версий`;
2. выбор версии;
3. понятный просмотр diff;
4. `Утвердить`;
5. `Отклонить`;
6. `Сделать текущей`;
7. `Открыть как рабочую`.

### Расширенный режим

Проверить:

1. branch creation;
2. merge;
3. что advanced actions не мешают обычному сценарию.

---

## 13. Практический вывод

Следующий этап для `TEXT REVISIONS` — это не расширение backend-функциональности, а **целенаправленный UX refactor**.

Приоритет реализации:

1. новый layout и терминология;
2. новый diff;
3. скрытие advanced workflow;
4. ручной UX smoke-check;
5. точечные правки по итогам ручного теста.
