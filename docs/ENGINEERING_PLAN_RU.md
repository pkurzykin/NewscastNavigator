# ENGINEERING PLAN — Newscast Navigator

## Цель

Привести проект к переносимой, воспроизводимой и устойчивой архитектуре без усложнения.

Проект должен:

* запускаться на любом сервере одинаково;
* не зависеть от конкретного окружения;
* не требовать ручных действий при деплое;
* быть готовым к постепенному росту.

---

## Базовые принципы

1. Не менять стек без необходимости
   (React + FastAPI + PostgreSQL + Docker — достаточен)

2. Web-only архитектура
   Streamlit и локальные сценарии не используются

3. Конфигурация только через environment variables

4. Код и данные строго разделены

5. Контейнеры считаются одноразовыми

6. Любая логика должна быть переносимой между серверами

---

## Этап 1 — обязательная подготовка (блокирует разработку)

До завершения этого этапа **разработка новых фич запрещена**.

### 1. Удалить хардкод

Запрещено:

* абсолютные пути
* IP / URL
* секреты
* CORS в коде

Всё через `.env`.

---

### 2. Настроить env-конфигурацию

Обязательно:

* `.env` (локально)
* `.env.example` (в репозитории)

Минимальный набор:

* DATABASE_URL
* SECRET_KEY
* CORS_ORIGINS
* STORAGE_PATH
* EXPORT_PATH

---

### 3. Разделить код и данные

Обязательно:

* база данных вне контейнера
* storage вне контейнера
* exports вне контейнера

Контейнер можно удалить без потери данных.

---

### 4. Единый запуск через Docker

Проект должен запускаться одной командой:

docker compose up

Без дополнительных ручных действий.

---

### 5. Миграции как единственный способ изменения БД

Запрещено:

* ручное изменение таблиц

Только Alembic.

---

### 6. Backup / Restore (минимум)

Должно быть:

* способ сделать backup БД
* способ восстановить БД

Минимально — через скрипты или команды.

---

### 7. Документация

Должно быть описано:

* как запустить проект
* как применить миграции
* как сделать backup/restore

---

## Критерий завершения Этапа 1

Можно переходить к разработке, если:

* проект поднимается с нуля через docker
* нет хардкода
* конфиг через env
* данные вне контейнеров
* миграции работают
* backup можно выполнить и восстановить

---

## Этап 2 — разработка

После подготовки:

### Правила

1. Не добавлять хардкод
2. Не привязываться к серверу
3. Не усложнять архитектуру
4. Не добавлять технологии без необходимости

---

### Проверка каждой фичи

Перед реализацией:

* переносимо ли это?
* не ломает ли docker?
* нет ли ручных действий?
* не смешаны ли код и данные?

Если есть проблема — исправить.

---

* очереди без необходимости
